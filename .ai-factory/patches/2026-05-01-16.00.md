# Fix hardcoded VS 2022 Professional path in build scripts

**Date:** 2026-05-01 16:00
**Files:** build-x64.bat, build-x86.bat, build-arm64.bat, build-debug-x64.bat, build-debug-win32.bat, build-winxp.bat, build-all.bat, find-vs.bat
**Severity:** medium

## Problem

All `.bat` build scripts hardcoded the path to `vcvarsall.bat` for Visual Studio 2022 **Professional** edition:
```
C:\Program Files\Microsoft Visual Studio\2022\Professional\VC\Auxiliary\Build\vcvarsall.bat
```

This failed on systems with:
- **Community** edition (most common free tier)
- **Enterprise** edition
- **BuildTools** edition
- Different install paths (e.g., `C:\Program Files\Microsoft Visual Studio\18\Community`)
- Non-default install directories

## Root Cause

The build scripts assumed all developers use VS 2022 Professional installed at the default path. Visual Studio supports multiple editions and can be installed at custom locations. The `vswhere.exe` utility (installed with every VS instance) exists precisely to solve this discovery problem, but the scripts didn't use it.

## Solution

1. **Created `find-vs.bat`** — a new helper script that locates `vcvarsall.bat` dynamically:
   - Primary: Uses `vswhere.exe` to query the latest VS installation with C++ tools
   - Fallback: Checks common paths for Enterprise → Professional → Community → BuildTools
   - Extra fallback: Scans `C:\Program Files\Microsoft Visual Studio\*` for non-standard paths
   - Sets `VCVARSALL` environment variable for the calling script

2. **Updated all 7 build scripts** — replaced the hardcoded path with:
   ```batch
   call "%~dp0find-vs.bat"
   if %errorlevel% neq 0 (
       echo ERROR: Failed to locate Visual Studio
       exit /b 1
   )
   call "%VCVARSALL%" <arch>
   ```

## Prevention

- Always use `vswhere.exe` or equivalent discovery tools instead of hardcoding IDE paths
- Test build scripts on clean machines with different VS editions before committing
- Document build prerequisites clearly (vswhere handles this transparently)
