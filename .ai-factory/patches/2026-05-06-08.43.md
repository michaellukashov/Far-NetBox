# Master password lockout message empty at runtime, stale stub warning, hardcoded recryption messages

**Date:** 2026-05-06 08:43
**Files:** src/resource/TextsWin.h, src/resource/TextsWin1.rc, src/NetBox/FarPluginStrings.cpp, src/windows/MasterPassword.cpp, src/windows/WinConfiguration.cpp, src/NetBox/WinSCPDialogs.cpp, src/base/MsgIDs.h, src/NetBox/NetBoxEng.lng, src/NetBox/NetBoxFr.lng, src/NetBox/NetBoxPol.lng, src/NetBox/NetBoxRus.lng, src/NetBox/NetBoxSpa.lng
**Severity:** high

## Problem

Verification of the master-password hardening plan revealed three runtime-affecting issues:

1. **Empty lockout message:** `ValidateMasterPassword()` threw an `Exception(FORMAT(LoadStr(NB_MASTER_PASSWORD_LOCKOUT).c_str(), ...))` but the lockout exception message was empty at runtime because `NB_MASTER_PASSWORD_LOCKOUT` (the NetBox enum value) was passed to `LoadStr()`, which tried to load a Win32 string resource by that numeric ID. No Win32 resource existed at that ID, so the string resolved to empty text.

2. **Stale stub warning:** `SecurityConfigurationDialog()` logged `WARNING: RecryptPasswords MSVC stub` even though `RecryptPasswords()` had already been implemented for MSVC in a prior commit.

3. **Hardcoded recryption messages:** `TMasterPasswordDialog::Execute()` displayed hardcoded English recryption error messages (`"Password changed, but %d session(s) could not be recrypted."` etc.) instead of using localized `GetMsg()` calls.

## Root Cause

**Issue 1 — Missing Win32 resource chain:** `LoadStr(int32_t Ident)` in `Common.cpp` calls `GetGlobals()->GetMsg(Ident)` which is `TGlobalFunctions::GetMsg`. This function first maps the ID through `FarPluginStrings[]` (Win32 ID → NetBox enum), then loads the localized string via the Far API from `.lng` files. When `LoadStr(NB_MASTER_PASSWORD_LOCKOUT)` was called, the NetBox enum value (~1362) was passed directly. There was no mapping for this value, so it fell through to `LoadStrFrom(InstanceHandle, 1362)` which loads a Win32 string resource by raw numeric ID. No such resource existed.

The correct pattern is: define a Win32 string ID (`MASTER_PASSWORD_LOCKOUT`), add it to `TextsWin1.rc`, map it in `FarPluginStrings.cpp`, and call `LoadStr(MASTER_PASSWORD_LOCKOUT)` so the mapping resolves to `NB_MASTER_PASSWORD_LOCKOUT` and loads from `.lng` files.

**Issue 2 — Leftover debug log:** The `AppLogFmt` warning was added before `RecryptPasswords()` was implemented. After implementation, the log became false and misleading.

**Issue 3 — Direct literals instead of GetMsg:** The recryption messages were added as inline `FORMAT(L"...")` calls. Since these are user-facing dialog messages in a dialog class (`TMasterPasswordDialog`), they should use `GetMsg(NB_...)` which loads localized strings from `.lng` files via the Far API.

## Solution

1. Added `#define MASTER_PASSWORD_LOCKOUT 7418` to `TextsWin.h`.
2. Added the English fallback string to `TextsWin1.rc`.
3. Added `{ MASTER_PASSWORD_LOCKOUT, NB_MASTER_PASSWORD_LOCKOUT }` mapping to `FarPluginStrings.cpp`.
4. Changed `LoadStr(NB_MASTER_PASSWORD_LOCKOUT)` to `LoadStr(MASTER_PASSWORD_LOCKOUT)` in both `MasterPassword.cpp` and `WinConfiguration.cpp`.
5. Removed the stale `AppLogFmt` stub warning from `WinSCPDialogs.cpp`.
6. Added `NB_MASTER_PASSWORD_RECRYPT_CHANGE`, `NB_MASTER_PASSWORD_RECRYPT_SET`, `NB_MASTER_PASSWORD_RECRYPT_CLEAR` to `MsgIDs.h`.
7. Added the three recryption strings to all 5 `.lng` files (English placeholders in non-English files, consistent with existing untranslated master-password strings).
8. Replaced hardcoded `FORMAT(L"Password changed...")` with `FORMAT(GetMsg(NB_MASTER_PASSWORD_RECRYPT_CHANGE).c_str(), ...)` and equivalents for set/clear.

## Prevention

- **When adding new `LoadStr()` calls:** Always verify the ID exists in the Win32 resource chain (`TextsWin.h` + `TextsWin1.rc`) AND has a `FarPluginStrings` mapping if it is a NetBox-specific string. A missing mapping causes silent empty-string fallback at runtime.
- **When implementing a stub:** Add a TODO or task to remove the stub warning once the implementation lands. The warning was missed because it was in a different file (`WinSCPDialogs.cpp`) from the actual implementation (`WinConfiguration.cpp`).
- **When adding user-facing dialog messages:** Always use `GetMsg()` (in dialog context) or `LoadStr()` (in backend context) instead of hardcoded literals. Follow the existing pattern in the same file.
- **After `/aif-implement` completes:** Run `/aif-verify` before considering the feature done. Verification caught these issues before they reached users.
